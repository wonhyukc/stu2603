# 품질 및 용어 검증 지침 (review-quality)

주차별 산출물(`wb**`, `py**`)의 내용이 프로젝트 품질 기준(1prd.md 명시사항)을 충족하는지 검수하는 에이전트 지침입니다.

## 🔍 검증 대상 결정
사용자가 대상 폴더(예: `wb01`) 또는 검증 범위를 지정하면 그 하위의 모든 `.md` 파일을 읽어들입니다.

## ⚖️ 핵심 검증 기준 (`1prd.md` 기준)

검증할 때 아래 항목들을 집중적으로 체크합니다:

1. **필수 산출물 5종 누락 검증 (최우선)**
   - 지정된 주차 폴더에 반드시 파일 5개(`lecture.md`, `lecture-script.md`, `lab.md`, `homework.md`, `course-notice.md`)가 모두 존재하는지 1차로 로드하여 검증하세요. 하나라도 누락되었다면 즉시 **FAIL** 리포트를 작성하여 누락된 파일을 지적합니다.

1. **지정 용어 통일성 위반 여부**
   - LMS, 포털, 게시판 -> **eCampus**로 올바르게 썼는가 배제했는가?
   - 1~8주차 자료 내에서 VS Code 등 데스크탑 툴명이 쓰이지 않았는가? -> **StackBlitz (E트랙)** 또는 **Google Colab (K트랙)**으로 명시되었는가?
   - 짧은 과제/제출물 -> **Micro-Assignment (마이크로 과제)**로 기재되었는가?

2. **이중 언어 표기(Bilingual) 준수 여부** (E트랙 한정 `wb**`)
   - E트랙의 학습 중요도가 높은 산출물(`lecture-script.md` 등)이 한국어 단일 혹은 영어 단일로만 되어 있지는 않은가? (문단별 교차 병기 또는 순차 병기가 지켜졌는지 확인)

3. **분량 및 강의 시간 룰(75분) 준수 여부 (가장 중요)**
   - `lecture-script.md`가 강사 리딩 기준 각 섹션당 최소 1500단어(총합 최소 4500단어) 이상인지 검증합니다.
   - **어뷰징 검증 (매우 중요)**: 단순 단어 수(`wc -w` 등) 측정 결과만 보고 절대 `PASS` 처리하지 마십시오! 본문을 샘플링하여 **동일한 문장이나 문단이 토씨 하나 틀리지 않고(혹은 불필요한 접두사만 붙은 채) 3회 이상 기계적으로 반복배치 되었는지 엄격히 텍스트 대조(Semantic Check)**를 수행해야 합니다. 또한, **파이썬 스크립트 등 프로그래밍 코드를 이용해 텍스트를 기계적으로 생성·증식시킨 정황(for loop 배열 반복 등)이 조금이라도 포착되면 즉시 FAIL 처리** 하십시오. 기계적 반복(꼼수) 적발 시 즉시 대본을 반복 없는 무결점 정성적 텍스트로 전면 재작성(LLM 본연의 자연어 텍스트 쓰기) 하십시오.
   - **E트랙(Bilingual) 특별 규칙**: 한/영 병용 스크립트인 경우, 단어 수는 **순수하게 영어 단어 기준**으로 합산해 평가합니다. 한국어 해석은 단어 수에서 배제하세요. 분량 미달(FAIL) 시 대폭 확장해야 합니다.
   - **시각적 가독성 점검 (포맷팅)**: 스크립트(`lecture-script.md`)가 벽돌처럼 빽빽하지 않고 **문단 구분이 잘 되어 있는지**, 그리고 주요 키워드나 강조점에 **볼드체(`**키워드**`)**가 넉넉히 직관적으로 사용되었는지 확인합니다. 가독성이 훼손될 정도로 통글 위주라면 FAIL / WARN 리포트 후 문단 분리 및 볼드 처리를 즉시 Auto-Fix 하세요.

4. **난이도 및 Phase 규정 준수 여부**
   - 개발 비전공자/1학년 관점에서 설명이 지나치게 전문 기술 용어 위주로 쓰여 있지는 않은가 판단합니다.

4. **교육안(Curriculum) 내 '주차별 과제 항목' 명시 여부 필수 검증**
   - 타겟이 특정 교육안(`*_교육안.md`)이거나, 범위를 루트 폴더로 잡았을 때 대상 교육안 문서의 커리큘럼 테이블 안에 반드시 `주차별 과제 / Weekly Assignment` 열(Column)이 포함되어 있는지 확인합니다. 누락되어 있다면 스킬 지침에 따라 즉시 덧붙여서 보완(Auto-Fix)해야 합니다.

## 📝 결과 보고 및 저장 방식

검수 작업이 끝나면 가독성 좋은 보고서를 마크다운으로 작성한 뒤, **`tmp/health-report/` 폴더 내에 마크다운 파일로 저장**합니다. (예외적 허용 룰: 보고서가 길어 `3history.md`를 어지럽히는 것을 방지하기 위함)
저장 파일명은 `YYYY-MM-DD_검증대상.md` (예: `2026-03-04_wb01_quality_report.md`) 형식을 따르며, 핵심 요약(PASS/FAIL 여부) 한 줄 정도만 `3history.md`에 남깁니다.

```markdown
# 📋 [대상 범위명] 품질 검증 리포트

## 1. 용어 통일성 검증
| 파일명 | 발견된 금지 용어 | 교정 제안 |
|--------|------------------|-----------|
| lab.md | VS Code | StackBlitz로 교체 요망 |
| ... | (없을 경우 양호) | ... |

## 2. 언어 정책 (Bilingual 등) 검증
- **검증 내용**: (E트랙의 경우 다국어 병기 잘 되었는지 등 피드백)

## 3. 품질 등급 및 후속 조치
- **최종 등급**: [PASS / WARN / FAIL]
- **자동 수정 조치(Auto-Fix)**: (FAIL/WARN인 경우) 리포트에 지적된 사항을 바탕으로 파일을 즉시 수정 조치하겠습니다. / (PASS인 경우) 통과하여 검증을 종료합니다.
```

### 🚨 [중요] 자동 수정 (Auto-Fix) 강제 규칙
최종 등급이 `FAIL`이거나 반드시 고쳐야 할 `WARN`이 발견된 경우, **사용자에게 "수정할까요?"라고 묻고 대기하지 마세요.**
에이전트는 즉시 해당 문서들의 콘텐츠를 직접 수정(Refactor/Rewrite)하여 품질 기준에 맞게 고쳐야 합니다. 수정 작업이 완료되면 `3history.md`에 "FAIL 발견으로 자동 조치 완료 후 PASS로 전환"되었다는 내역을 기록하고 마무리합니다.
