# 프로젝트 이슈 및 브랜치 워크플로우 가이드

프로젝트 요구사항 문서(`1prd.md`)에 명시된 규칙에 따라, 이슈를 확인하고 코드를 작성한 뒤 최종 반영(Merge)하기까지의 과정을 5단계로 나누어 설명합니다.

## 🔄 이슈부터 변경사항 반영까지의 5단계 워크플로우

### 1. 작업할 이슈 확인 (세션 시작)
작업을 시작하기 전에 목표가 담긴 '이슈(Issue)'를 먼저 확인합니다.
- 터미널에서 `gh issue list` 명령어를 입력하거나 GitHub 웹사이트에서 열려 있는 이슈들을 확인합니다.
- 내가 처리할 이슈를 선택하고, 작업에 착수하겠다는 상태를 보고한 뒤 작업을 시작합니다.

### 2. 전용 작업 공간(브랜치) 만들기
메인(`main`) 브랜치에 직접 코드를 작성(직접 커밋)하는 것은 금지되어 있습니다. 안전하게 작업하기 위한 분리된 공간을 만들어야 합니다.
- 이슈 번호와 간단한 설명을 조합하여 새로운 브랜치를 생성합니다.
- **이름 규칙**: `issue-{이슈번호}-{짧은-설명}`
- **예시**: `issue-12-add-week1-lecture` (12번 이슈: 1주차 강의안 추가)

### 3. 작업 후 저장하기 (커밋, Commit)
코드를 작성하거나 문서를 수정한 후에는 내역을 저장(커밋)합니다.
- **[가장 중요한 규칙]**: 권장 정책에 따라 커밋 메시지는 **무조건 한글**로만 작성합니다.
- **형식**: `[타입] 한글로 작성한 작업 내용 (#이슈번호)`
- **타입 종류**:
  - `feat`: 새로운 기능/자료 추가
  - `fix`: 오류 및 버그 수정
  - `docs`: 문서 작업
  - `refactor`: 기능 변경 없는 코드 개선
  - `chore`: 잡일, 설정 변경 등
- **결과 예시**: `[feat] 1주차 파이썬 강의안 작성 (#12)`

### 4. 작업물 반영 요청 (PR, Pull Request)
나의 작업 브랜치를 원래의 `main` 브랜치에 합쳐 달라고 요청(PR)하는 단계입니다.
- PR의 내용(또는 제목) 어딘가에 `Closes #{이슈번호}` 라는 문구를 반드시 포함합니다.
- **이유**: 기재해 두면 PR이 메인에 성공적으로 합쳐질 때, 해당 이슈가 자동으로 '완료(Closed)' 처리되어 관리가 편해집니다.

### 5. 최종 반영 (병합, Merge)
PR 내용에 문제가 없는지 확인이 끝나면, 변경 사항을 `main`에 합치게 됩니다.
- 병합할 때는 여러 번의 자잘한 커밋 내역을 깔끔하게 하나로 뭉쳐서 합치는 **Squash merge**(스쿼시 머지) 방식을 사용해야 합니다.

---

## 💡 핵심 요약

1. **할 일 확인 필요함(Issue)**
2. ➔ **내 작업복 입기(Branch)**
3. ➔ **한글로 작업 일지 쓰기(Commit)**
4. ➔ **결제 올리기(PR + `Closes #번호`)**
5. ➔ **깔끔하게 서류 합치기(Squash Merge)**
