# 프로젝트 이슈 및 브랜치 워크플로우 가이드

프로젝트 요구사항 문서(`1prd.md`)에 명시된 규칙에 따라, 이슈를 확인하고 코드를 작성한 뒤 최종 반영(Merge)하기까지의 과정을 5단계로 나누어 설명합니다.

## 🔄 이슈부터 변경사항 반영까지의 5단계 워크플로우

### 1. 작업할 이슈 확인 (세션 시작)
작업을 시작하기 전에 목표가 담긴 '이슈(Issue)'를 먼저 확인합니다.
- 터미널에서 `gh issue list` 명령어를 입력하거나 GitHub 웹사이트에서 열려 있는 이슈들을 확인합니다.
- 내가 처리할 이슈를 선택하고, 작업에 착수하겠다는 상태를 보고한 뒤 작업을 시작합니다.

### 2. 전용 작업 공간(브랜치) 만들기
메인(`main`) 브랜치에 직접 코드를 작성(직접 커밋)하는 것은 금지되어 있습니다. 안전하게 작업하기 위한 분리된 공간을 만들어야 합니다.
- 이슈 번호와 간단한 설명을 조합하여 새로운 브랜치를 생성합니다.
- **이름 규칙**: `issue-{이슈번호}-{짧은-설명}`
- **예시**: `issue-12-add-week1-lecture` (12번 이슈: 1주차 강의안 추가)

### 3. 작업 후 저장하기 (커밋, Commit)
코드를 작성하거나 문서를 수정한 후에는 내역을 저장(커밋)합니다.
- **[가장 중요한 규칙]**: 권장 정책에 따라 커밋 메시지는 **무조건 한글**로만 작성합니다.
- **형식**: `[타입] 한글로 작성한 작업 내용 (#이슈번호)`
- **타입 종류**:
  - `feat`: 새로운 기능/자료 추가
  - `fix`: 오류 및 버그 수정
  - `docs`: 문서 작업
  - `refactor`: 기능 변경 없는 코드 개선
  - `chore`: 잡일, 설정 변경 등
- **결과 예시**: `[feat] 1주차 파이썬 강의안 작성 (#12)`

### 4. 작업물 반영 요청 (PR, Pull Request)
나의 작업 브랜치를 원래의 `main` 브랜치에 합쳐 달라고 요청(PR)하는 단계입니다.
- PR의 내용(또는 제목) 어딘가에 `Closes #{이슈번호}` 라는 문구를 반드시 포함합니다.
- **이유**: 기재해 두면 PR이 메인에 성공적으로 합쳐질 때, 해당 이슈가 자동으로 '완료(Closed)' 처리되어 관리가 편해집니다.

### 5. 최종 반영 (병합, Merge)
PR 내용에 문제가 없는지 확인이 끝나면, 변경 사항을 `main`에 합치게 됩니다.
- 병합할 때는 여러 번의 자잘한 커밋 내역을 깔끔하게 하나로 뭉쳐서 합치는 **Squash merge**(스쿼시 머지) 방식을 사용해야 합니다.

---

## 💡 핵심 요약

1. **할 일 확인 필요함(Issue)**
2. ➔ **내 작업복 입기(Branch)**
3. ➔ **한글로 작업 일지 쓰기(Commit)**
4. ➔ **결제 올리기(PR + `Closes #번호`)**
5. ➔ **깔끔하게 서류 합치기(Squash Merge)**

## 🎓 학기 초 vs 매주: 생성 자동화 가이드라인 (course-work & build-week)
프로젝트 내 에이전트를 어떻게 활용하여 스킬과 워크플로우를 다룰 수 있는지, 시기별 올바른 사용법을 함께 안내해 드립니다.

### 1단계: 전체 뼈대 잡기 (학기 초 기획 단계)
* **적용 스킬/워크플로우**: `@[course-work]`
* **수행하는 일**: 1주차부터 15주차까지 전체 커리큘럼(큰 그림)을 기획합니다.
* **산출물**: 각 트랙의 마스터 파일(`[트랙]_교육안.md`)에 "주차별 학습 목표와 주요 내용"이 세팅됩니다.

### 2단계: 실무 자료 찍어내기 (매주 수업 전)
* **적용 스킬/워크플로우**: `@[build-week]` (또는 `/주차생성 3`)
* **수행하는 일**: 위에서 만들어둔 마스터 뼈대(`[트랙]_교육안.md`)를 바탕으로, 해당 주에 당장 강의실에서 써야 할 **디테일한 3종 세트**를 뽑아냅니다.
* **산출물**:
  1. `lecture.md` (강사용 75분 강의안 & 코딩 대본)
  2. `lab.md` (학생용 실습 가이드)
  3. `homework.md` (과제 지시서)

### 3단계: 바람직한 매주 자동화 지시 방법 (프롬프트 예시)
에이전트에게 2주차(또는 N주차) 생성을 지시할 때는 한 번에 모든 것을 시키기보다는 스킬, 워크플로우, 검증 단계를 **차근차근 순차적으로** 지시하는 것이 가장 오류가 적고 확실합니다. 아래 3단계를 순서대로 명령해 보세요.

#### Step 1: 자료 생성 요청 (Skill)
> `@[build-week] 스킬 지침에 따라 py01 폴더에 2주차 파이썬 강의 자료(lecture, lab, homework)를 모두 생성해줘. '강사 리딩 기준 75분(25분 x 3개 섹션)' 규칙을 명확히 지켜서 분량을 조절해.`

#### Step 2: 교차 검증 요청 (Skill)
> `@[review-quality] 스킬을 사용하여 방금 생성된 py01 주차 자료들이 1prd.md의 지시사항(트랙별 언어 규칙, 난이도)과 75분 분량 룰 및 용어 통일성을 잘 지켰는지 검증해 줘. 부족한 부분이 있다면 보완해 줘.`

#### Step 3: 병합 및 마무리 (Workflow)
> `작업과 검증이 모두 끝났으니, @[/commit-clean] 워크플로우를 실행해서 이번 2주차 자료 생성 내역만 깔끔하게 커밋으로 마무리해 줘.`
